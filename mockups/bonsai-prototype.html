Bonsai v1 — UX Prototype

Clickable prototype exploring how the first version of Bonsai delivers to users. Walk the four core flows: launch → new project → onboard existing project → daily use. Click a card to enter that flow.

A. Run Bonsai
Flow A1

Launch / Project picker

What the user sees the first time they open Bonsai after `./run.sh`. Recent projects, "open existing", "start new".
B. Create new project
Flow B1

Chat-first new-project session

Agent interviews the user to draft the Product Document live in the right pane. Progressive disclosure as fields fill in.
C. Open existing project
Flow C1

Onboarding (no Bonsai yet)

`/spec-from-code` session — agent scans repo, reads CLAUDE.md/AGENTS.md, drafts modules & product summary for review.
Flow C2

Project home (Bonsai initialized)

Spec tree (left) · open document (center) · adaptive context dashboard (right) — drift, coverage, recent tickets.
D. Daily — tickets & sessions
Flow D1

New ticket — ChatGPT-style

Big "+" → centered input. Agent picks the right skill: clarifies if vague, drafts immediately if clear.
Flow D2

Ticket detail (session container)

Vertical timeline of phases — describe → specify → plan → execute. Each phase holds the sessions that produced it.
Flow D3

Adaptive right rail (ticket focus)

Same home layout — but with a ticket selected, the right rail switches from project dashboard to ticket-relevant context.
Flow D4

Tickets — connected view

All tickets shown with relationships (depends-on, follows, blocks). Lightweight graph layout.
Reading guide

Each page has a floating navigator at the bottom — use it to jump between flows. The prototype is intentionally static (no JS state): every interaction is a separate page so we can compare states side-by-side.

Arborist

A reading-list app with zero-friction capture and a clean offline reader.

Goals

  • Capture any URL from any device in two taps.
  • Read offline, with my own pace, without distractions.
  • Sync state (read/unread, position, highlights) across all my devices.

Principles

  • Respect attention. No notifications, no social feed, no recommendation engine.
  • Local-first. The reader works fully offline; sync is a background convenience.
  • Self-hostable. All state lives on hardware I own.

Audience

  • Single user (me) for v1. Multi-user is explicitly out of scope.

Stack & Constraints

  • TypeScript end-to-end (web + extension + iOS share sheet via Capacitor).
  • SQLite locally. Sync server is a small Go binary on a self-hosted box.
  • Auth: a single long-lived JWT per device — no third-party auth providers.
  • Hard no: any cloud service that can't be replaced in a weekend.

task-spec module: capture git: a8ae469

iOS share-sheet capture

Capture any URL from the iOS share sheet without leaving the source app.

Behavior

  • Tapping "Send to Arborist" in the share sheet enqueues the URL into the local SQLite store.
  • If the device is offline, the URL is queued and uploaded when sync next runs.
  • Capture must complete (UI dismissal) within 250ms even when offline.

Out of scope

  • Highlighting, annotation, or pre-fetch — those belong to the reader module.

Acceptance

  • Manual: capture a URL from Safari while in airplane mode → URL appears in the local list within 1s, syncs once online.
  • Test: capture/share-sheet.test.ts covers the queue + retry path.

3. The session may ask 1–3 clarifying questions, then writes a clean ticket description.
4. The ticket lands on your home in described state — ready to specify, plan, and execute.

Why a graph?

Tickets often spawn each other — a feature spec uncovers a bug, a refactor unlocks a new capability. The graph lets you see how today's work connects to past decisions and future intent. Solid arrows = explicit dependency, dashed = follows / suggested.
