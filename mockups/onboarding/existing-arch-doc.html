payments-api · architecture

FastAPI service routing payments across Stripe and PayPal, backed by a Postgres ledger. Drafted from existing code — confirm before saving.

Stack

  • Runtime: Python 3.11 · async FastAPI 0.110.
  • Provider SDKs: stripe-python 7, paypalrestsdk 1.13.
  • Storage: Postgres + Alembic migrations.
  • Tests: pytest, 62 test files, mock provider fixtures.

Source: pyproject.toml, app/main.py, tests/.

Modules

Each module gets a stub specs/<name>/README.md on save — see right rail.

app/providers/

Stripe + PayPal adapters. Well-tested. PCI scope — raw card data must never leave this module.

  • Public surface: StripeProvider, PayPalProvider, both implement Provider.charge(req)
  • Depends on: vendor SDKs only · zero internal deps

app/router/

Chooses provider per transaction by fee. Includes a fallback to next-cheapest on decline.

  • Public surface: choose_provider(charge), charge(req)
  • Depends on: providers, ledger.record_attempt()
  • ⚠ Health-skip logic exists (healthcheck()) but isn't wired — see Known gaps.

app/webhooks/

Receives provider callbacks. Verifies signature per provider. Idempotent by (provider, event_id).

  • Public surface: POST /webhooks/{provider}
  • Depends on: ledger.apply_event(), vendor signature libs

app/ledger/

Append-only internal accounting. Source of truth for all attempts and outcomes.

  • Public surface: record_attempt(), apply_event(), balance(merchant_id)
  • Depends on: Postgres only
  • ⚠ Test coverage ~30% · candidate for follow-up ticket.

Data flow

  1. Charge. POST /v1/charge → router.choose_provider() → provider.charge() → ledger.record_attempt().
  2. Decline → fallback. Router walks remaining providers (next-cheapest first) until one accepts or all decline.
  3. Webhook. Provider POSTs to /webhooks/{provider} → signature verified → dedup by (provider, event_id) → ledger.apply_event().

Design principles

  • PCI containment. Raw card data lives only in providers/; tokenize before crossing module boundaries.
  • Append-only ledger. Corrections are new entries, not mutations. Audit trail is sacred.
  • Idempotency over speed. Better to ack a webhook safely twice than to double-charge once.
  • Vendor neutrality. Adding a provider should not require changes outside providers/ + a small registration in router/__init__.py.

⚠ Known architectural gaps

  • Provider health-skip not wired. healthcheck() exists but never called from route.py. Required by G&R · Must have #2.
  • Ledger module test coverage ≈ 30%. Risky for the source-of-truth module.
