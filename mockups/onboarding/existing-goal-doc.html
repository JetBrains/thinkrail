payments-api

Route every payment to the cheapest viable provider per transaction, with PCI-compliant tokenization and idempotent webhook intake.

Context

payments-api is an internal payment-processing service for the e-commerce platform. It sits behind the order-service and abstracts over multiple providers (Stripe and PayPal), choosing one per transaction. Card data is tokenized inside provider modules and never crosses module boundaries in raw form.

Source: README.md, CLAUDE.md, app/main.py, app/providers/.

Problem

Each provider charges different per-transaction fees that depend on currency, card brand, and volume tier. Without routing, payments default to a single provider and the company over-pays. Webhooks from providers need idempotent intake — replays must not double-credit.

Audience

  • Internal callers — the order-service via /v1/charge.
  • External partners (in progress) — /v2/charge exposes rate-limiting + API-keys for partner integrators. Treat partner-readiness as a constraint going forward.

Scope

What this project is

  • Per-transaction routing across Stripe and PayPal.
  • PCI-compliant tokenization in provider modules.
  • Idempotent webhook intake by (provider, event_id).
  • An internal ledger that is the source of truth.

Out of scope

  • Payouts, refunds workflows beyond raw provider passthrough.
  • Customer-facing UI — that lives in the storefront.
  • Fraud detection (assumed handled upstream).

Requirements

Must have

#RequirementRationale
1Route by per-transaction fee for the given currency & card brand.Core goal of the service.
2Skip providers whose last 5 health checks failed. known gapFrom clarification: cost-only routing has caused outages.
3Tokenize cards inside provider modules; never log raw PAN.PCI scope.
4Webhooks idempotent by (provider, event_id).Provider replays are normal; double-credits are not.
5Ledger entries are append-only; corrections are new entries.Audit trail for accounting.

Constraints

  • Python 3.11, FastAPI 0.110, stripe-python 7, paypalrestsdk 1.13. Postgres via Alembic.
  • PCI scope is limited to app/providers/; other modules must not handle raw card data.
  • v2 endpoints must remain backwards-compatible once a partner is live.
  • Webhook handler must ack the provider within 10s — slow downstream work belongs on a queue.

Known gaps (found while reading)

  • Provider health-skip not wired. healthcheck() exists but isn't called by route.py. Confirmed as intended-but-incomplete.
  • Ledger test coverage ≈ 30% — likely worth a follow-up ticket.
  • Router fallback logic undocumented — captured in this doc now.
