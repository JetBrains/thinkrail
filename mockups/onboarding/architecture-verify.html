rookery · architecture

Mobile-first React Native client with offline-first SQLite, syncing to a small FastAPI/Postgres backend.

Stack

  • Client: React Native (Expo), TypeScript.
  • On-device storage: SQLite (expo-sqlite).
  • Backend: FastAPI on Python 3.11, async.
  • Server storage: Postgres + Alembic migrations.
  • Media: CDN-fronted blob store; thumbnails on-device.

Modules

capture

Owns the act of recording a sighting. Single entry point, no UI navigation logic.

  • Public surface: captureSighting(input) → Sighting
  • Depends on: sync.queue(), device camera/GPS APIs

sync

Owns the offline queue and reconciliation with the backend. Eventually consistent.

  • Public surface: queue(item), flush(), reconcile()
  • Depends on: SQLite, FastAPI /v1/sightings

lifelist

Owns the read side: chronological feed, map view, life-list count. Read-only against SQLite.

  • Public surface: useLifelist(), useMap(), useCount()
  • Depends on: SQLite (read-only)

Data flow

  1. User taps Log → capture.captureSighting() writes a Sighting row to SQLite immediately.
  2. sync.queue() appends to the outbox table.
  3. When online, sync.flush() POSTs queued items to FastAPI; on success, marks them synced.
  4. lifelist reads from SQLite — sees both synced and unsynced rows transparently.

Design principles

  • Local-first. SQLite is source of truth on-device; the backend is a sync target, not a critical path.
  • Separation of concerns. capture only writes; lifelist only reads; sync bridges them.
  • Explicit over implicit. Sighting unsynced state is a visible flag in the row, not a hidden detail.
