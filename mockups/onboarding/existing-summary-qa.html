payments-api

Route every payment to the cheapest viable provider per transaction, with PCI-compliant tokenization and idempotent webhook intake.

Context

payments-api is an internal payment-processing service. It sits behind order-service and abstracts over Stripe and PayPal, choosing one per transaction. Card data is tokenized inside provider modules and never crosses module boundaries in raw form.

Problem

Each provider charges different per-transaction fees. Without routing, payments default to one provider and the company over-pays. Webhooks need idempotent intake — replays must not double-credit.

Audience

  • Internal callers — order-service via /v1/charge.

Scope · in

  • Per-transaction routing across Stripe and PayPal.
  • PCI-compliant tokenization in provider modules.
  • Idempotent webhook intake by (provider, event_id).
  • Internal ledger as source of truth.

Requirements · must have + added

  • Route by per-transaction fee for the given currency & card brand.
  • Skip providers whose last 5 health checks failed. known gap
  • Tokenize cards inside provider modules; never log raw PAN.
  • Webhooks idempotent by (provider, event_id).
  • Ledger entries are append-only.

Constraints

  • Python 3.11, FastAPI, stripe 7, paypalrestsdk 1.13.
  • PCI scope limited to app/providers/.
  • Webhook ack within 10s.

Known gaps

  • Provider health-skip not wired (healthcheck() never called from route.py).
  • Ledger test coverage ≈ 30%.
  • Router fallback logic was undocumented (now captured here).
