Implement Spec service.py: facade for all spec CRUD operations

service.py is the single entry point for all spec operations. It is called from
RPC methods (user CRUD) and watcher callbacks (disk change detection). It
delegates to parser, validator, graph, and registry — no direct file or registry
access happens outside this facade.

Spec reference: backend/app/spec/README.md

Public interface to implement:

- list_specs() → list[SpecSummary]
    Read registry, return lightweight summaries. May return empty list.

- get_spec(id: str) → SpecDetail
    Lookup entry in registry, parse file from disk, assemble SpecDetail with links.
    Raises SpecNotFoundError if ID not in registry or file missing on disk.

- create_spec(type: str, path: str, content: str | None = None) → SpecDetail
    Create spec file on disk + add registry entry.
    Title is auto-derived from first heading in content (or from path if no content).
    ID is generated (slug from path or title). Status defaults to "draft".
    Raises on path conflict or invalid type.

- update_spec(id: str, content: str) → SpecDetail
    Update spec file content on disk + update registry entry (updated timestamp).
    Raises SpecNotFoundError or validation failure.

- delete_spec(id: str) → None
    Remove spec file from disk + remove registry entry.
    Raises SpecNotFoundError if not found.

- get_graph() → SpecGraph
    Read registry, build and return full hierarchy graph.

Dependencies:
